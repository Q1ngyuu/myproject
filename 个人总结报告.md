# 实训个人总结报告

**Blog System — Next.js + Flask 全栈博客系统**

---

## 一、遇到的最大困难及解决过程

优化阶段最头疼的问题，是返回首页时滚动位置和分页状态会丢。从文章详情页点返回，首页直接滚回顶部的轮播图，分页也从第 4 页跳回第 1 页，体验很差。

我的办法是用 sessionStorage：离开首页时把页码和滚动标记存下来，回来再恢复。思路本身不复杂，但测试时发现一个怪现象——页面像刷新了两次，第一次明明回到了第 4 页，紧接着又跳回第 1 页。一开始完全没头绪，后来开着 DevTools 一步步跟，才定位到是 SearchBar 组件里防抖的 useEffect 在挂载时触发了一次 onSearch("")，Home 的 handleSearch 收到后执行 setPage(1)，把刚恢复的页码又盖掉了。

第一版修复用的是 isFirstRender ref，想跳过首次渲染，结果没生效。又查了一阵才发现根因在 React 18 的 Strict Mode：开发模式下组件会挂载、卸载、再挂载，而 ref 在两次挂载之间不会重置。第一次挂载时 isFirstRender 已经变成 false，第二次挂载检查就失效了，onSearch("") 照样触发。最后改成 userInteracted ref，只有用户真的输入内容或点了清除才置 true，跟组件挂载几次彻底无关，问题这才算解决。

这个 bug 折腾了我挺久，但收获也最大。最大的教训是：改 bug 之前得先把背后的机制搞清楚，急着动手的话，修出来的东西很可能本身就是新 bug。

## 二、对 AI 辅助编程的心得体会

这次实训里，AI 从写文章内容、改文档到排查 bug 基本都参与了，用下来有惊喜也有教训。

先说惊喜。AI 干重复性的活是真的快。22 篇文章扩充加上两份文档更新，都是大量结构化文本，AI 直接生成初稿，我逐篇审查修改就行，效率至少提了五倍，省下的时间可以花在架构和 bug 上。排查问题的时候，AI 又像一只会回话的橡皮鸭：我把现象描述给它，它给出几个假设，我挨个验证，再把结果喂回去，几轮下来排查范围就缩得很小了。

教训也有。第一次那个 isFirstRender 方案，其实是我和 AI 一起定下来的，结果两边都没想到 Strict Mode 会让它失效。所以 AI 给的代码一定要自己验证，尤其是涉及框架机制的地方，它说得再头头是道也不能直接照搬，项目整体的架构判断更得自己拿主意。

用下来的感受是：AI 把"怎么写"这一步变得很快，但"写什么""为什么这么写""这么写对不对"，还是得人来决定。会用 AI 不是让它替你写代码，而是自己先有判断，再用它把活干快。

## 三、个人贡献

- **文章内容扩充**：把 22 篇短文扩成内容详实的 Markdown 长文（每篇 80–150 行），技术类覆盖 Docker、Redis、CSS Grid、TypeScript 泛型、Next.js SSR 等 12 个主题，另外 10 篇是生活随笔。backend/add_posts.py 和 frontend/src/lib/store.ts 两个数据源同步更新，净增 3600 多行代码。

- **项目文档更新**：对照项目实际状态把 README.md 和 API.md 过了一遍，修正了分类和文章数量的错误，补上漏掉的组件和脚本，新增了双后端架构说明和分页、搜索参数的文档。

- **前端体验优化**：实现了返回首页自动滚到文章列表区域（跳过轮播图）；用 sessionStorage 持久化分页和搜索状态，返回时能恢复到离开前的页码。

- **Strict Mode bug 修复**：定位并解决了 SearchBar 在双重挂载下 onSearch 误触发的问题，查清了 isFirstRender 方案失效的根因，最终改用 userInteracted ref 彻底修复。
